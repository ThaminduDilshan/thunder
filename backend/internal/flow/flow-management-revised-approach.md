# Flow Management API - Revised Approach

## Problem Statement

The initial approach had a **dual-representation model** where:
- Frontend sends step-based UI definitions
- Backend translates to graph representation
- Creates complexity in translation logic and maintaining two representations

See: [flow-management-implementation.md](./flow-management-implementation.md) for the initial approach details.

## New Approach: Single Graph Representation

### Key Changes from Initial Approach

| Aspect | Initial Approach | Revised Approach |
|--------|------------------|------------------|
| **Representation** | Dual (Steps + Graph) | Single (Graph only) |
| **Translation** | Backend translates steps → graph | No translation needed |
| **Storage** | Two representations in DB (`step_definition` + `graph_definition`) | Single representation (`graph_definition` only) |
| **Frontend Responsibility** | Send step-based UI definitions | Construct complete graph with all nodes |
| **Decision Nodes** | Auto-generated by backend | Explicitly added by frontend |
| **Node Types** | VIEW, EXECUTION, special steps | UI_INTERACTION, TASK_EXECUTION, DECISION, special nodes |
| **Layout** | Size + Position in steps | Top-level `layout` with size/position for all nodes |
| **Complexity** | High (translator, dual storage, sync) | Low (validate and store as-is) |

### Core Principles

1. **Frontend and Backend share the same graph representation** (JSON format)
2. **Introduce a new UI node type** (`UI_INTERACTION`) to encapsulate UI metadata
3. **No translation layer needed** - what frontend sends is what backend stores
4. **Frontend is responsible for graph construction** including decision nodes
5. **Single source of truth** - one format for management, storage, and execution

### Architecture Overview

```
┌──────────────────────────────────────────────────────────────┐
│                    Flow Composer UI (Frontend)               │
│  - Visual drag-and-drop interface                            │
│  - Constructs complete graph with UI nodes                   │
│  - Inserts decision nodes when multiple options exist        │
└────────────────────┬─────────────────────────────────────────┘
                     │ Graph JSON
                     ▼
┌──────────────────────────────────────────────────────────────┐
│              Flow Management API (Backend)                   │
│  - Stores graph definition as-is (no translation)            │
│  - Validates graph structure                                 │
│  - Serves graph for runtime execution                        │
└────────────────────┬─────────────────────────────────────────┘
                     │ Same Graph JSON
                     ▼
┌──────────────────────────────────────────────────────────────┐
│              Flow Execution Engine (Backend)                 │
│  - Loads graph from database                                 │
│  - Executes nodes including UI nodes                         │
│  - Returns UI metadata when UI node is encountered           │
└──────────────────────────────────────────────────────────────┘
```

## New Node Type: UI_INTERACTION

**Decided**: Use `UI_INTERACTION` as the node type name.
- Clear purpose: handles user interaction
- Consistent with existing naming (TASK_EXECUTION, AUTHENTICATION_SUCCESS)
- Can be renamed later if needed

### Node Layout Requirements

**All nodes** must have top-level `layout` property for flow composer positioning:
- **layout.size**: { width, height } - dimensions of the node box in the composer
- **layout.position**: { x, y } - where the node is placed in the canvas
- Users can drag and resize nodes in the flow composer UI

**UIDefinition**: Only UI_INTERACTION nodes have `uiDefinition` property containing `components` array. Other node types (TASK_EXECUTION, DECISION, AUTHENTICATION_SUCCESS, etc.) do not have `uiDefinition`.

**Note**: End-user UI layout/styling (for rendered login/registration forms) is handled separately via the branding component, not in the flow definition.

## Design Considerations

### 1. Graph Structure

All nodes in the flow share a common structure with:
- **ID**: Unique identifier within the flow
- **Type**: Node type (UI_INTERACTION, TASK_EXECUTION, DECISION, etc.)
- **Layout**: Required for all nodes at top level - size and position for flow composer visualization
- **Properties**: Optional executor-specific properties (e.g., idpId)
- **UIDefinition**: Only for UI_INTERACTION nodes - contains `components` array with UI components
- **Executor**: Executor definition (for TASK_EXECUTION nodes)
- **Next**: Array of next node IDs

### 2. Node Types

- **UI_INTERACTION**: New node type that encapsulates UI components and user interaction logic
- **TASK_EXECUTION**: Executes backend logic (existing)
- **DECISION**: Routes flow based on conditions (existing)
- **AUTHENTICATION_SUCCESS**: Terminal node for successful authentication
- **REGISTRATION_START**: Starting point for registration flows
- **PROVISIONING**: User provisioning node for registration

### 3. Layout System

**Layout at Top Level**: All nodes have top-level `layout` property with:
- **size**: { width, height } - Node dimensions in flow composer
- **position**: { x, y } - Node placement on canvas

**UIDefinition**: Only UI_INTERACTION nodes have `uiDefinition` property containing `components` array. Other node types do not have `uiDefinition`.

**Note**: End-user UI styling (for login/registration forms) is handled separately via the branding component.

### 4. Action Handling

Actions in UI_INTERACTION nodes:
- Each ACTION component has an `action` object with:
  - `type`: "NEXT" (for navigation)
  - `next`: Target node ID
- **Single action**: Direct connection to target node
- **Multiple actions**: Requires DECISION node to route between options

### 5. Validation Requirements

The graph validator ensures:
- Node IDs are unique
- All `next` references point to valid nodes
- UI_INTERACTION nodes have at least one ACTION component
- Multiple actions require DECISION node as next node
- DECISION node includes all action targets in its `next` array
- All nodes have valid layout (size and position)
- Flow type-specific requirements (e.g., REGISTRATION needs PROVISIONING)

### 6. Storage Model

- **Single representation**: Only graph stored in database
- **No translation**: What frontend sends is what backend stores
- **Database**: Single `graph_definition` JSONB/TEXT column
- **Simplicity**: No dual storage, no sync issues
- **Layout structure**: All nodes have top-level `layout`; only UI_INTERACTION nodes have `uiDefinition`

## Benefits of Revised Approach

1. **Simplicity**: No translation layer, no dual representation
2. **Consistency**: Frontend and backend always in sync (single graph format)
3. **Flexibility**: Frontend has full control over graph structure
4. **Performance**: No compilation overhead during save (store as-is)
5. **Maintainability**: Single source of truth for flow definition
6. **Debugging**: Easier to trace issues (what you see in UI is what's stored in DB)
7. **Extensibility**: Easy to add new node types or properties to graph
8. **Clear Separation**: Single layout structure (top-level `layout`) for flow composer positioning, `uiDefinition` reserved for UI components only
9. **Backend Simplification**: Flow management service just validates and stores (no complex translation logic)

## Important Notes

### Internationalization (i18n)

Component text values can be either:
- **Plain text**: `"Login"`, `"Enter your username"`
- **i18n keys**: `"{{ t(auth:login.title) }}"`, `"{{ t(registration:username.label) }}"`

**Syntax**: `{{ t(namespace:key) }}`

**Backend handling**: 
- Backend does NOT process or validate i18n syntax
- Text values are stored as-is in the graph
- Frontend is responsible for parsing and resolving i18n keys
- This is purely a frontend concern for rendering

### Node-Specific Executors

- **REGISTRATION_START**: No executor needed - handled automatically by backend
- **PROVISIONING**: Requires `ProvisioningExecutor`
- **AUTHENTICATION_SUCCESS**: Requires `AuthAssertExecutor` (for registration flows)
- **TASK_EXECUTION**: Requires executor name (e.g., `BasicAuthExecutor`, `GoogleOIDCAuthExecutor`)

### Layout Requirements

- **All nodes** must have top-level `layout` property with `size` and `position`
- This is for flow composer visualization only
- **uiDefinition** is only for UI_INTERACTION nodes and contains `components` array
- Other node types (TASK_EXECUTION, DECISION, etc.) do not have `uiDefinition`
- End-user UI styling is handled via branding component (separate from flow definition)

## Component Reference

This section provides a brief reference for UI components used in flow definitions.

### Component Structure

All components have:
- **id**: Unique identifier within the node
- **category**: Component category (see table below)
- **type**: Component type (see table below)
- **variant**: Optional variant for specific styling/behavior
- **config**: Configuration object (varies by component type)
- **components**: Child components (for BLOCK types only)
- **action**: Action definition (for ACTION types only)

### Valid Component Combinations

| Category | Type | Variants | Purpose | Required in FORM |
|----------|------|----------|---------|------------------|
| **DISPLAY** | TYPOGRAPHY | H1, H2, H3, H4, BODY, CAPTION | Display text | No |
| **DISPLAY** | DIVIDER | - | Visual separator with optional text | No |
| **DISPLAY** | IMAGE | - | Display images | No |
| **FIELD** | INPUT | TEXT, PASSWORD, EMAIL, TEL, NUMBER | User input fields | Yes |
| **FIELD** | CHECKBOX | - | Boolean selection | Yes |
| **FIELD** | RADIO | - | Single selection from options | Yes |
| **FIELD** | SELECT | - | Dropdown selection | Yes |
| **ACTION** | BUTTON | PRIMARY, SECONDARY, SOCIAL, TEXT | Trigger actions (submit/navigate) | Yes (if submitting form) |
| **ACTION** | LINK | - | Navigation links | No |
| **BLOCK** | FORM | - | Groups inputs for submission | No |
| **BLOCK** | TABS | - | Tabbed interface with conditional content | No |
| **BLOCK** | CARD | - | Visual grouping | No |

### Important Rules

1. **FIELD components must be inside a FORM** to be submitted
2. **ACTION (BUTTON with submit type) must be inside a FORM** to submit form data
3. **FORM components** can contain FIELD, ACTION, and DISPLAY components
4. **TABS components** can contain any components; use `visibleWhen` config to show/hide based on active tab
5. **DISPLAY components** can be placed anywhere (inside or outside forms)
6. **ACTION components with type="button"** (not submit) can be outside forms for navigation

### Example Form Structure

```json
{
  "id": "form_login",
  "category": "BLOCK",
  "type": "FORM",
  "config": {},
  "components": [
    {
      "id": "input_username",
      "category": "FIELD",
      "type": "INPUT",
      "variant": "TEXT",
      "config": { "identifier": "username", "required": true }
    },
    {
      "id": "btn_submit",
      "category": "ACTION",
      "type": "BUTTON",
      "variant": "PRIMARY",
      "config": { "type": "submit", "text": "Login" },
      "action": { "type": "NEXT", "next": "auth_task" }
    }
  ]
}
```

## Flow Definition Examples

This section provides comprehensive examples for various authentication scenarios, showing both the visual flow structure and complete graph definitions using the revised approach (single graph representation with UI_INTERACTION nodes).

### Important Note on SMS/Email OTP Flows

**Revised Approach Limitation**: The revised graph-based approach currently does not support having two nodes with the same executor in a single flow (e.g., two `SMSOTPAuthExecutor` nodes for send and verify). This is because:
- SMS/Email OTP requires two separate interactions: collect mobile/email → send OTP, then collect OTP code → verify OTP
- Each interaction needs a separate executor invocation
- The revised examples show only the **first step** (mobile/email collection before sending OTP)

**Initial Step-Based Approach**: This limitation does NOT exist in the initial approach because:
- Frontend sends two VIEW steps: Step 1 (mobile input) → Step 2 (OTP input with SMS executor)
- Backend translator constructs the graph during execution: PROMPT_ONLY → TASK_EXECUTION (SMS) → PROVISIONING
- Translation layer handles the complexity of multiple interactions

**Future Solution for Revised Approach**: Requires architectural changes:
- Separate "Send OTP" and "Verify OTP" executors, OR
- Stateful executor that handles both send and verify phases, OR
- Support for multiple nodes with the same executor type

## Summary

### Key Decisions

| Decision Point | Finalized Approach |
|----------------|-------------------|
| **Representation** | Single graph format (no dual representation or translation) |
| **Node Type Name** | `UI_INTERACTION` for UI nodes |
| **Layout Structure** | Top-level `layout` with size/position for ALL nodes (flow composer positioning only) |
| **Action Type** | `action.type = "NEXT"` with `action.next` pointing to target node ID |
| **i18n Format** | `{{ t(namespace:key) }}` syntax (not validated by backend) |
| **Multiple Actions** | Requires DECISION node; validated by backend |
| **Storage** | Single `graph_definition` column in database |
| **REGISTRATION_START** | No executor needed (handled by backend) |
| **Success Messages** | Use separate UI_INTERACTION node if needed (not in AUTHENTICATION_SUCCESS) |

### What Changed from Initial Approach

- **No translation layer**: Frontend constructs graph directly (not steps → graph)
- **No dual storage**: Single representation in database
- **Simpler backend**: Validate and store as-is (no compilation)
- **Frontend responsibility**: Insert DECISION nodes when multiple actions exist
- **Layout structure**: All nodes have top-level `layout`; only UI_INTERACTION nodes have `uiDefinition`
